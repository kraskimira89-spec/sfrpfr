# ТЗ: база знаний кейсов и RAG (без дообучения)

## Принцип

Не дообучать модель на переписках. Диалоги могут содержать ошибки, неподтверждённые выводы и ПДн.

Правильная схема:

```text
Диалоги DeepSeek → очистка ПДн → разбор на кейсы → экспертная проверка
→ база знаний → RAG-поиск → AI-помощник → обратная связь эксперта
```

Модель получает только подходящие **проверенные** фрагменты прошлых кейсов перед ответом.

## Что извлекаем из диалога

- запрос клиента и исходная ситуация;
- документы и их типы;
- найденные расхождения: стаж, ИЛС, ФИО, СНИЛС, работодатель;
- применённый алгоритм;
- подготовленные заявления и запросы;
- итог СФР: удовлетворено / отказ / не завершено;
- фактический финансовый результат (диапазоны);
- оценка эксперта: «подход верный», «ошибка», «нужна доработка».

## Ограничение по ПДн

Перед загрузкой в RAG / другой AI / таблицы удалить или заменить:

- ФИО, телефон, адрес;
- СНИЛС, паспорт;
- сканы документов;
- номера выплатных дел;
- точные даты рождения;
- ссылки на личные кабинеты и файлы.

Идентификатор кейса в базе знаний: `CASE-YYYY-NNN` (например `CASE-2026-001`).

## Реестр обезличенных кейсов

Поля:

| Поле | Описание |
|---|---|
| `case_id` | `CASE-YYYY-NNN` |
| `problem_type` | тип проблемы |
| `documents` | типы документов |
| `discrepancy` | найденное расхождение |
| `legal_basis` | основание / норма |
| `prepared` | что подготовлено |
| `sfr_outcome` | результат СФР |
| `pension_increase_range` | рост пенсии (диапазон) |
| `edv_range` | ЕДВ (диапазон) |
| `expert_assessment` | оценка эксперта |
| `verified_at` | дата проверки |
| `quality` | статус качества |

Хранение MVP: `knowledge/cases/*.json` (+ опционально краткий `.md` для чтения).

## Статусы качества

| Статус | Смысл |
|---|---|
| `draft` | извлечено из переписки, не проверено |
| `verified` | эксперт подтвердил |
| `rejected` | неверный или неполный подход |
| `template` | типовой сценарий для RAG |

В RAG попадают **только** `verified` и `template`.

## Контур улучшения

После дела эксперт отмечает:

- какие документы реально понадобились;
- что сработало / не сработало;
- итоговый ответ СФР;
- можно ли использовать кейс как шаблон (`quality=template`).

## CLI

```text
sfrfr knowledge-depersonalize-dir inbox/ --out cleaned/
sfrfr knowledge-import-deepseek conversations.json -n 5 --cleaned-dir storage/knowledge_inbox/cleaned
sfrfr knowledge-import PATH     # диалог → draft JSON (с обезличиванием)
sfrfr knowledge-list            # реестр
sfrfr knowledge-set-status ID Q # draft|verified|rejected|template
```

`knowledge-depersonalize-dir`: md/txt/json/html/csv → очищенные копии; PDF/сканы пропускаются.  
`knowledge-import-deepseek`: официальный экспорт DeepSeek (`conversations.json`) → пенсионные диалоги по title → draft. `user.json` не импортировать (там email/id).

## Системный промпт AI-помощника

См. `ASSISTANT_SYSTEM` в `src/sfrfr/ai/prompts/__init__.py`.

Кратко: анализировать только документы + проверенную базу; не обещать выплат; не выдумывать нормы; отделять факт от предположения; не повторять лишние ПДн; клиент подаёт в СФР самостоятельно.

## Критерии приёмки

- Импорт всегда проходит через обезличивание ПДн.
- `draft` / `rejected` не участвуют в RAG-поиске.
- Черновик заявления помечается `needs_human_review=True`.
- В git не попадают сырые диалоги с ПДн (только обезличенные кейсы).
