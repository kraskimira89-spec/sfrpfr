# ТЗ-09: паритет мини-приложения MAX и веб-кабинета клиента

## Цель

Клиент выбирает канал работы с делом:

1. **мини-приложение MAX** (`/app/` внутри бота);
2. **веб-кабинет** (`cabinet.домен`).

Одинаковые функции дела доступны в обоих каналах. Источник истины — один: FastAPI + Supabase. WordPress только приводит к выбору канала, ПДн и файлы туда не попадают.

## Для кого ТЗ

Кодер (Agent mode): API, `apps/cabinet`, `web/max-miniapp`, лендинг/бот, миграции.

## Текущее состояние (исходная точка)

| Канал | Стек | Что умеет сейчас | API |
|---|---|---|---|
| Мини-приложение | `web/max-miniapp` | открыть дело, статус, upload, `run`, findings | `/api/cases/*`, `/api/documents/upload` (MAX `user_id`) |
| Веб-кабинет | `apps/cabinet` | OTP, список дел, карточка, согласия, оплаты, результат, сообщения | `/api/portal/*` (JWT) |

Разрыв: два контура API и разная глубина функций. Клиент не выбирает канал явно; связка MAX ↔ аккаунт кабинета слабая или отсутствует.

## Принципы

1. **Один `case_id` — два UI.** Смена канала не создаёт второе дело.
2. **Один контракт данных.** Оба UI читают/пишут через согласованный client API (предпочтительно расширенный `/api/portal` + привязка MAX).
3. **Паритет функций MVP**, не паритет пикселей. Визуал может отличаться; сценарии — нет.
4. **Выбор канала — продуктовый шаг**, не технический костыль.
5. **Без кнопки «подать в СФР за клиента»** в обоих UI.
6. **Предупреждение** на ключевых экранах: «Решение принимает СФР. Результат не гарантирован».

## 1. Выбор канала клиентом

### 1.1. Точки выбора

| Место | Поведение |
|---|---|
| WordPress CTA «Начать проверку» | Экран/блок **«Как вам удобнее?»**: MAX или веб-кабинет |
| Бот MAX (первый контакт / `/start`) | Кнопки: «Открыть мини-приложение» и «Ссылка на веб-кабинет» |
| Мини-приложение (шапка/меню) | Ссылка «Открыть в браузере» → `cabinet.домен` с deep-link на то же дело |
| Веб-кабинет (шапка/меню) | Блок «Продолжить в MAX» → deeplink бота / mini-app |
| Email/SMS уведомления | Две ссылки: кабинет + MAX (если `max_user_id` известен) |

### 1.2. Предпочтение канала

Хранить у клиента (или у связи client↔case):

```text
preferred_channel: max_miniapp | web_cabinet | unset
preferred_channel_set_at: timestamptz
```

- При первом явном выборе — сохранить.
- Клиент может сменить в настройках кабинета / мини-приложения.
- Не блокировать другой канал: предпочтение влияет на CTA и уведомления по умолчанию, не на доступ.

### 1.3. Копирайт выбора (коротко)

- «В MAX — быстро из мессенджера»
- «В браузере — удобнее с компьютера и большим экраном»
- Оба варианта: одни и те же документы и статус дела

## 2. Матрица паритета функций (MVP)

Обязательный паритет — статус **Must** в обоих каналах.

| Функция | Мини-приложение | Веб-кабинет | Примечание |
|---|---|---|---|
| Вход / идентификация | MAX WebApp `user_id` + связь с клиентом | OTP email/телефон (Supabase) | После линка — один профиль |
| Согласие на ПДн до upload | Must | Must | Одна версия текста/версии |
| Создать / открыть своё дело | Must | Must | Один `case_id` |
| Список моих дел | Must (если >1) | Must | |
| Карточка: статус pipeline | Must | Must | Одинаковые RU-лейблы этапов |
| Ближайшее действие / чек-лист | Must | Must | |
| Загрузка PDF/JPG/PNG | Must | Must | private Storage + audit |
| Список загруженных документов | Must | Must | без публичных URL |
| Запуск проверки (`run` / advance) | Must | Must | или «запросить проверку» → очередь эксперта — единообразно |
| Findings (кратко) | Must | Must | |
| Черновик заявления (чтение) | Must | Must | без автоподачи в СФР |
| Инструкция подачи в СФР/МФЦ/Госуслуги | Must | Must | |
| Сообщения по делу | Must (или deep-link в чат бота) | Must | если в mini-app нет треда — явная кнопка «написать в чат» |
| Согласия / оферта / заказ | Should | Must | mini-app: ссылки на юр. страницы + акцепт |
| Оплаты | Should | Must | mini-app может открывать webview/внешнюю оплату |

**Out of scope этого ТЗ:** функции admin/expert (`admin.домен`), RAG knowledge review.

## 3. Модель идентичности и связка каналов

### 3.1. Сущности

```text
clients
  id
  full_name
  phone / email
  preferred_channel
  max_user_id          -- nullable, unique
  auth user_id         -- Supabase auth.users, nullable до первого OTP

cases
  id
  client_id
  ...существующие поля...
```

### 3.2. Сценарии линка

**A. Сначала MAX**

1. Клиент открывает mini-app → создаётся/находится дело по `max_user_id`.
2. В mini-app или боте: «Привязать веб-кабинет» → OTP на телефон/email.
3. После OTP: `clients.max_user_id` и `clients.user_id` связаны.
4. Веб-кабинет показывает то же дело.

**B. Сначала веб**

1. OTP → кабинет → дело.
2. «Открыть в MAX» → бот `/start` с payload `case_id` или login token.
3. При первом сообщении/открытии mini-app — привязка `max_user_id` к `client_id` (с подтверждением OTP, если уже есть другой max_user).

**C. Конфликт**

- Один `max_user_id` не может принадлежать двум clients.
- При конфликте — сообщение оператору + клиенту «уже привязано»; без автослияния ПДн.

### 3.3. API связки (минимум)

```text
POST /api/portal/auth/otp/request
POST /api/portal/auth/otp/verify
POST /api/portal/link/max          -- JWT + max_user_id / initData
POST /api/portal/link/web-from-max -- из mini-app: старт OTP
GET  /api/portal/me                -- профиль + preferred_channel + linked flags
PATCH /api/portal/me/preferences   -- preferred_channel
```

Проверка подписи MAX WebApp `initData` — на сервере (не доверять голому `user_id` из query в проде).

## 4. Единый client API (целевой контракт)

Мини-приложение постепенно переводится с «голого» `/api/cases` на portal-совместимый контур.

### 4.1. Обязательные методы для обоих UI

```text
GET  /api/portal/me/cases
GET  /api/portal/cases/{id}
POST /api/portal/cases                    -- создать дело (если нет)
POST /api/portal/cases/{id}/documents
GET  /api/portal/cases/{id}/documents
POST /api/portal/cases/{id}/run           -- или /request-review (единая семантика)
GET  /api/portal/cases/{id}/findings
GET  /api/portal/cases/{id}/draft
GET  /api/portal/cases/{id}/checklist
GET|POST messages, consents, orders, result  -- как в текущем portal
```

Авторизация:

- веб: `Authorization: Bearer <JWT>`;
- mini-app: заголовок `X-MAX-InitData` (или обмен initData → short-lived portal token).

### 4.2. Deprecation

- `/api/cases/open` для mini-app — допустим в переходный период, но новые фичи только в portal.
- После паритета: mini-app не пишет в отдельный in-memory store без Supabase.

## 5. Задачи по фронтам

### 5.1. `apps/cabinet` (веб)

Добавить/довести до Must из матрицы всё, что уже в mini-app и ещё нет в вебе:

- кнопка **«Запустить проверку»** (или эквивалент `run` / request-review);
- блок findings;
- явный **переключатель «Продолжить в MAX»**;
- экран/блок настроек канала (`preferred_channel`);
- deep-link `cabinet.домен/cases/{id}` из писем и из mini-app.

Сохранить: OTP, согласия, оплаты, результат, сообщения.

### 5.2. `web/max-miniapp`

Довести до Must:

- список дел (если несколько);
- чек-лист / ближайшее действие;
- просмотр черновика заявления;
- инструкция подачи;
- согласие с версией до upload;
- ссылка **«Открыть веб-кабинет»** (с token/deep-link);
- по Should: оплаты и результат — допустим переход в webview кабинета на те же URL.

Вынести общие RU-лейблы статусов в общий модуль/JSON (избежать расхождений).

### 5.3. Бот MAX

- `/start`: выбор канала + открытие mini-app;
- команда/кнопка «Веб-кабинет» с персональной ссылкой (magic link / OTP landing);
- уведомления о смене статуса — с двумя ссылками, приоритет по `preferred_channel`.

### 5.4. WordPress

- CTA ведёт на страницу/якорь выбора канала (не только в MAX).
- По-прежнему без upload СНИЛС/сканов на WP.

## 6. UX-ограничения

- Первый экран после входа: статус + следующее действие + CTA загрузки/проверки (без перегруза).
- На мобильном web-кабинете — usable; mini-app остаётся primary для «из чата».
- Одинаковые тексты предупреждений и статусов.
- Ошибки API показывать человекочитаемо в обоих UI.

---

## 7. Безопасность и ПДн

- RLS: клиент видит только свои `case_id`.
- Signed URL с TTL для файлов в обоих каналах.
- Audit log: upload, consent, link max↔web, смена preferred_channel, run.
- Не логировать полный СНИЛС; в mini-app по-прежнему только маска, если поле нужно.
- CORS: `cabinet.домен` + origin витрины `/app/` + при необходимости MAX webview origin.
- Запрет прямой выдачи service role в браузер.

## 8. Этапы реализации для кодера

### Этап A — связка и предпочтение (фундамент) ✅ реализовано

1. Миграция: `max_user_id`, `preferred_channel` у clients.
2. `GET/PATCH /api/portal/me` (+ preferences).
3. Link MAX↔web (OTP + initData verify).
4. CTA выбора на WP и в боте.

### Этап B — паритет операций по делу ✅ реализовано

1. Portal: `run` / findings / draft / checklist единым контрактом.
2. Веб-кабинет: кнопки проверки, findings, deep-link в MAX.
3. Mini-app: чек-лист, draft, согласие, ссылка в веб; CaseRead с status_label/hint.

3. Mini-app на portal API (`X-MAX-InitData`); deep-link кабинета `?case=&view=`.

## 9. Критерии приёмки

1. Клиент может создать/вести **одно** дело и открыть его и в MAX mini-app, и в `cabinet.домен` после привязки.
2. Upload документа в одном канале виден в другом после refresh ≤ N секунд (или сразу из БД).
3. Статусы и findings совпадают в обоих UI.
4. Выбор канала сохраняется и влияет на CTA уведомлений.
5. Смена канала не требует повторной загрузки всех документов.
6. Нет возможности подать заявление в СФР от имени клиента ни в одном UI.
7. Чужой `case_id` недоступен (403/404) в обоих каналах.
8. На витрине есть явный выбор MAX vs веб, не только одна кнопка в мессенджер.
9. Автотесты: link max↔web; upload ACL; preferences patch.

## 10. Тест-план (минимум)

- [ ] WP → выбор MAX → mini-app → дело → upload → статус
- [ ] WP → выбор веб → OTP → то же дело после link из MAX
- [ ] Mini-app → «Открыть в браузере» → OTP → то же `case_id`
- [ ] Веб → «Продолжить в MAX» → mini-app видит документы
- [ ] `run`/проверка из веба отражается в mini-app и наоборот
- [ ] Конфликт: второй auth не перехватывает чужой `max_user_id`
- [ ] Preferred channel меняет текст/порядок ссылок в уведомлении-заглушке
