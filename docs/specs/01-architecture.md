# ТЗ: общая архитектура SFRFR

## Цель

Разделить публичный сайт и контуры обработки персональных данных. Клиентские документы и данные пенсионных дел не должны попадать в WordPress.

## Схема

```text
site.домен      → WordPress: витрина, услуги, оферта, FAQ
cabinet.домен   → клиентский кабинет
admin.домен     → внутренний кабинет сотрудников
api.домен       → FastAPI API

Supabase        → Auth, PostgreSQL, Storage, RLS
Taganay CRM     → лиды, воронка, задачи
MAX Bot API     → диалог с клиентом и уведомления
Yandex AI Studio→ AI-агенты
Google Sheets   → только обезличенная аналитика
```

## Доменные зоны

| Домен | Назначение | Доступ |
|---|---|---|
| `site.домен` | WordPress | публичный |
| `cabinet.домен` | кабинет клиента | авторизованные клиент и представитель |
| `admin.домен` | кабинет сотрудников | оператор, эксперт, администратор |
| `api.домен` | FastAPI, Swagger и webhook MAX | приложения, MAX, разработчики |

## Ответственность компонентов

- WordPress не хранит документы, СНИЛС, паспортные данные и ключи API.
- FastAPI реализует бизнес-логику, аудит ИЛС и трудовой, интеграции и выдачу временных ссылок.
- Supabase хранит учётные записи, данные кейсов и приватные файлы.
- Кабинеты используют API; прямой доступ браузера к служебным ключам запрещён.
- MAX используется для диалога, уведомлений и **мини-приложения**; источник истины по делу — FastAPI и Supabase.
- Клиент может работать в **мини-приложении MAX** или в **веб-кабинете** (`cabinet.домен`) с одним `case_id` — см. [09-client-channels-parity.md](09-client-channels-parity.md).

## Поток дела

```text
intake → documents_received → ocr_done → classified → extracted
→ audited → draft_ready → human_review → completed
```

Статус `failed` применяется при технической ошибке. Финальное решение по заявлению всегда принимает СФР; автоматическая подача от имени клиента не реализуется.
