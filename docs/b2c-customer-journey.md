# Клиентский путь B2C

**Версия:** 0.1  
**Дата:** 2026-07-22  
**См. также:** [b2c-monetization-tz.md](b2c-monetization-tz.md), [architecture.md](architecture.md)

## 1. Каналы входа

- Лендинг WordPress (кнопка в MAX / форма заявки).
- Бот MAX → webhook API.
- Звонок / мессенджер → ручной ввод в CRM Taganay → `case_id` в SFRFR.

## 2. Карта статусов

### 2.1. Операционный пайплайн (существующий)

`intake` → `documents_received` → `ocr_done` → `classified` → `extracted` →  
`audited` → `draft_ready` → `human_review` → `completed` (+ `failed`)

### 2.2. Финансовые / клиентские статусы (B2C)

| Статус | Смысл |
|--------|--------|
| `lead` | Заявка создана, в CRM |
| `consent_accepted` | Согласие на ПДн |
| `diagnostic_paid` | Оплачена диагностика |
| `contract_accepted` | Акцепт оферты + индивидуальный заказ |
| `service_paid` | Оплачено сопровождение |
| `package_delivered` | Пакет и инструкция выданы клиенту |
| `awaiting_client_submission` | Клиент подаёт сам |
| `result_pending` | Ждём решение / доказательства |
| `result_confirmed` | Эксперт подтвердил результат |
| `success_fee_due` | Наступил срок постоплаты (2–3 мес.) |
| `success_fee_paid` | Постоплата получена |
| `client_silent_escalation` | 6 мес. без ответа |
| `closed` | Дело закрыто |

## 3. Пошаговый journey

| Шаг | Действие клиента | Система | CRM Taganay | Ответственный |
|-----|------------------|---------|-------------|---------------|
| 1 | Оставляет заявку | Создаёт lead + `case_id` | Стадия «Новый лид» | Оператор |
| 2 | Принимает согласие ПДн | `consent_accepted` | Задача «собрать документы» | Клиент |
| 3 | Оплачивает диагностику | `diagnostic_paid`, чек | Стадия «Диагностика» | Клиент |
| 4 | Загружает документы | `documents_received` → OCR… | Чек-лист задач | Клиент / эксперт |
| 5 | Получает план и чек-лист | `audited` + checklist | Обновление задач | Эксперт |
| 6 | Акцептует индивидуальный заказ | `contract_accepted` | Стадия «Сопровождение» | Клиент |
| 7 | Оплачивает сопровождение | `service_paid` | Оплата зафиксирована | Клиент |
| 8 | Получает пакет документов | `package_delivered` | Задача «подача клиентом» | Эксперт |
| 9 | Подаёт в СФР/МФЦ/Госуслуги | `awaiting_client_submission` | Контроль срока | Клиент |
| 10 | Загружает решение | `result_pending` | Задача проверки | Клиент |
| 11 | — | Эксперт подтверждает → `result_confirmed` | Стадия «Результат» | Эксперт |
| 12 | Через 2–3 мес. | `success_fee_due` + счёт | Стадия «Постоплата» | Система |
| 13 | Оплачивает SF | `success_fee_paid` → `closed` | «Успех» | Клиент |
| 14* | Молчит 6 мес. | `client_silent_escalation` | «Эскалация» | Юрист / оператор |

\* Альтернативная ветка.

## 4. Индивидуальный чек-лист

Создаётся после первичного аудита, уникален для дела.

Минимальные поля пункта:

- `title` — что сделать / какой документ;
- `type` — document | action | archive_request | review;
- `owner` — client | expert;
- `due_at` — срок;
- `status` — open | done | blocked | cancelled;
- `note` — комментарий без лишних ПДн в Sheets.

Примеры пунктов: «Запросить архивную справку по ООО …», «Сверить период 1998–2001», «Подать заявление через Госуслуги», «Загрузить решение СФР».

## 5. Уведомления

| Событие | Канал | Содержание |
|---------|-------|------------|
| Оплата / чек | MAX / email | Ссылка, сумма |
| Готов чек-лист | MAX | Список недостающего |
| Пакет готов | MAX / кабинет | Файлы + инструкция подачи |
| Напоминание о подаче | MAX | Срок |
| Запрос результата | MAX | Через N дней после подачи |
| Счёт success fee | MAX / email | Сумма, срок 2–3 мес. окна |
| 30 / 90 / 150 / 180 дней без ответа | MAX / email | Эскалация к 6 месяцам |

`first_contact_at` фиксируется при шаге 1 и не меняется.

## 6. Google Sheets (дубль без ПДн)

При смене этапа CRM / дела — upsert строки:

`case_id | segment | region_bucket | stage | problem_type | created_at | paid_diag | paid_service | result_band | sf_due | sf_paid | silent_flag`

Секретные и персональные поля **не** синхронизируются.
